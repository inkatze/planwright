- 2026-07-23 [planwright] fleet-streamjson.sh concurrency-hardening follow-up (panel lens findings, declined in the task-4 polish loop as lower-incidence for single-supervisor-per-worker usage, narrow windows, alarm-scan backstopped): (1) journal.lock stale-break has no ownership token, so two waiters past the 60s threshold can both break-and-take, admitting concurrent holders that can lose a receipt; (2) recover.lock has no stale-break, so a SIGKILLed foreground recover wedges recovery permanently; (3) attention_settled/upsert run outside the journal lock, so a settle-clear can race a concurrent append and briefly drop a queue item (alarm-scan re-surfaces it at the threshold); (4) pid files are not validated against process start-time, so pid reuse after a crash corrupts status/recover/answer liveness verdicts; (5) the answer path opens the stdin fifo >> while holding the journal lock, so a worker death in the check->open TOCTOU can block the open and stall the lock until the 60s break. Consider: an ownership-token/pid-stamp on both locks, moving attention writes under the lock, a non-blocking fifo open (O_NONBLOCK) in answer, and start-time-checked liveness. Seeded for a hardening spec-draft.
