- 2026-07-23 [planwright] fleet-status.sh render pads columns with printf '%-20s ...', which counts BYTES, so non-ASCII worker-authored strings misalign that row's columns. Two distinct behaviors, split by the sanitizer's C1 range: names whose UTF-8 continuation bytes are all >= 0xA0 (U+00E0-U+00FF lowercase accented Latin as C3 A0..C3 BF, and e.g. U+4E2D) pass through sanitize_printable intact and merely shift columns; names carrying a continuation byte in 0x80-0x9F (every U+00C0-U+00DF uppercase accented Latin as C3 80..C3 9F, plus much CJK/Hangul/symbols) have those bytes DELETED under the pinned C locale, so they render as mojibake with a dangling lead byte — verified: 'CAFE-worker' with an acute E renders as CAF<?>-worker. The same 0x80-0x9F rule makes fleet-liveness oracle --list drop such a session row whole. Both are the deliberate security-over-fidelity trade echo-safety.sh documents, repo-wide and pre-dating this PR, not a status-view regression. CLI-render-only: the tab-separated merge stream (Task 8's contract) is unaffected, and nothing is hidden or truncated by the padding itself. Reviewed 2026-07-23 during copilot-review on PR #315 and deliberately left as-is: a wcwidth-aware fix in POSIX sh is disproportionate, and the mangling belongs to the shared sanitizer, not to this renderer. Task 8's HTML/browser dashboard sidesteps byte-padding natively (the browser lays out text), but it inherits the sanitizer's mangling if it renders sanitized values rather than encoding raw ones for HTML.
