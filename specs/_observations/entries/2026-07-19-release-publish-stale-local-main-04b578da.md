- 2026-07-19 [planwright] release-publish.sh derives the release version from LOCAL main (MAIN_REF=main), but the orchestrator never advances local main (merges land on origin/main via GitHub PRs), so mise run release run from an orchestration checkout computes the STALE version and refuses on the idempotency gate (tag vX.Y.Z already exists / already published) even though origin/main carries the newer bumped version. Hit during the v0.23.0 release 2026-07-19: PR #235 (release 0.23.0) merged on origin, manifest 0.23.0 on origin/main, but local main sat at 0.22.0, so release-publish computed v0.22.0 and refused repeatedly; release-pending.sh (also reading local main) falsely reported none. Fix required a manual git -C <repo> merge --ff-only origin/main before the release published. Deterministic fix for release-hardening: release-publish/release-pending should read the version-of-truth from a fetched origin/main, or the flow should fetch and ff local main as a pre-step, so a stale local main cannot mis-derive the pending version. Distinct from the known cancelled-CI deadlock; this is a stale-ref version-derivation bug. Feeds the release-hardening spec.
