- 2026-07-19 [planwright] Worker idle/attention detection should be HOOK-based, not tower pane-polling. Root problem (bit us 2026-07-19): the tower polls each worker tmux pane with capture-pane and grep-heuristics to guess busy-vs-idle-vs-fork. This false-NEGATIVED a stuck fork on Task 2 for ~hours — the busy-detector grepped the whole captured pane and scrollback text (Waiting for N background agent, spinner words) masked the actual idle-at-fork footer, so the tower never fired an attention event and the worker sat blocked at a real CI-config sign-off. Deterministic fix = a worker-side Claude Code Stop hook (fires when the turn ends / goes idle) and/or Notification hook (fires when the session needs attention: permission prompt, idle-waiting) that PUSHES a signal the instant it happens — e.g. writes a per-worker attention fragment (fleet-state/attention/<task> with reason+timestamp) that the tower watches via Monitor/inotify. Zero poll latency, no pane-parsing, no scrollback false-match. This is the SAME hook mechanism as the PreToolUse permission auto-approve hook this very spec (worker-permission-ergonomics) builds — hooks are the deterministic answer for BOTH the permission flood AND attention detection, and this is the fleet-autonomy attention-surface (D-9) done right. Caveats: needs the hook wired into worker launch settings (planwright never edits settings.json, so a shipped worker-attention-hook + wiring instruction, like the worker-settings profile); cannot retrofit already-running workers (hooks load at session start). Stopgap applied this session: footer-only busy/attention detection (grep last ~10 pane lines, not the whole scrollback). Connects to tower-hook-extension-amendment and idle-detect-background-agent-fp.
