# planwright options reference

The single canonical reference for every planwright config option (REQ-K1.8,
D-43). The tracked defaults live in [`config/defaults.yml`](../config/defaults.yml);
per-repo overrides live in `<repo>/.claude/planwright.local.yml` (gitignored,
agent-maintained, per-repo entries written only on human confirmation per
REQ-K1.1).

Every option present in the default config must have a row here.
`scripts/check-options-reference.sh` enforces that; planwright's CI runs it
(wired in by the self-hosting CI task) and fails on an undocumented option.

Config-model fallback: a skill that finds a config file absent, unreadable,
or malformed falls back to the tracked defaults below and surfaces a
warning when it reads the config, before any default-driven action fires
(REQ-K1.7).

Format constraints the checker relies on: each row's first table cell must
contain only the backticked option name (annotations go in the Effect
column), and `config/defaults.yml` must stay flat `key: value` lines (nested
YAML keys would be invisible to the parser, which fails closed when it
parses zero keys).

| Option | Default | Effect | Consumed by |
| --- | --- | --- | --- |
| `commit_on_draft` | `true` | Auto-commit the completed Draft bundle after elicitation finishes (D-41). Commit only, never push. | `/spec-draft` |
| `commit_on_kickoff` | `true` | Auto-commit the kickoff brief plus the Draft→Ready status flip after sign-off (D-41). Commit only; the push, draft PR, and terminal ready-flip are separate kickoff steps. | `/spec-kickoff` |
| `commit_on_state_move` | `true` | **v1-only.** Auto-commit `tasks.md` orchestration-state edits — section moves (e.g. `/drain` relocating a block between human-owned sections) and the per-block `Status` / `Last activity` annotations (e.g. `/execute-task` stamping `Last activity`) — with a fixed conventional message (D-41). Vacuous under format-version 2: a v2 bundle commits no derived execution-state edit (the version-keyed machinery no-ops those writes, D-7), so there is nothing to auto-commit and the knob is not consulted. Status is read through the derived render, not a committed snapshot (see [orchestration state](orchestration-state.md)). | `/drain`, `/execute-task` |
| `mark_spec_pr_ready_on_kickoff` | `true` | On a clean kickoff completion, `/spec-kickoff` marks the **spec** PR ready (un-draft) as its terminal step, after the configurable `review_sequence` verification has converged (D-6, D-7) — the narrow exception to bootstrap D-26's all-drafts rule: only the spec PR is readied, task PRs stay drafts, merge stays the human's (never auto-merge). `false` leaves the spec PR draft for the human to un-draft. The flip is also suppressed when sign-off parked on a fork or verification did not converge; a flip failure degrades to `tasks.md` Awaiting input (REQ-D1.2, REQ-D1.3). | `/spec-kickoff` |
| `dispatch_backend` | `subagent` | Which dispatch backend `/orchestrate` uses for execution units: `subagent` (default), `tmux`, `print`, or `in-session` (D-38, REQ-F1.8). Each value maps to an advertised capability set per the [backend capability contract](../doctrine/backend-capability-contract.md) (D-2), which defines how a new terminal/multiplexer becomes a selectable value by advertising the contract rather than by a skill edit. `/orchestrate` autodetects the backends actually present and, attended, presents them and asks (never a silent pick); unattended, it reads this value and, when that backend is absent or not autonomously selectable, degrades down the degradation ladder — never to a silently-chosen interactive backend (REQ-B1.4, D-3). So this value is the configured default and the unattended selection input; the per-backend dispatch adaptation is still name-keyed pending a later task. | `/orchestrate` |
| `dispatch_isolation` | `per-step` | How `/execute-task` sequences a unit's steps (D-5, REQ-C1.3): `per-step` (default) runs implementation and each configured `review_sequence` skill in its own fresh `/resume`-seeded session, so context stays bounded and each review's perspective is uncontaminated by prior steps (backends that cannot spawn a fresh session approximate it via context clears between steps); `per-unit` keeps today's single-session behavior for the whole unit. The `per-step` default is the assigned human decision; `per-unit` stays supported for constrained hosts. Resolved through `scripts/resolve-dispatch-isolation.sh`; a value other than `per-step`/`per-unit` is malformed under the REQ-E1.4 by-layer policy (degrade+warn for adopter/machine-local, hard-fail for repo-tracked). The default-flip from the historical `per-unit` behavior is tracked cross-spec as the required bootstrap D-38 amendment. | `/execute-task` |
| `context_budget_threshold` | `50` | When a long-running tower auto-heals (D-4, REQ-C1.1): a completed-step budget — the number of orchestration steps a tower runs before it hands off to a fresh tower (the [continue-as-new](../doctrine/context-budget-autoheal.md) pattern), so it never silently degrades from context exhaustion. The signal is a step-count proxy because Claude Code exposes no supported live context-usage introspection (the transcript is internal/unstable; no env var, hook field, or CLI query for remaining budget — Task 5 research). `off` disables auto-heal (the historical single-tower behavior). The default is conservative because the handover is cheap and lossless (the fresh tower rebuilds from durable state), so handing off early is the safe direction. Resolved through `scripts/resolve-context-budget-threshold.sh`; a value other than a positive integer of at most 15 digits or `off` is malformed under the REQ-E1.4 by-layer policy (degrade+warn for adopter/machine-local, hard-fail for repo-tracked) — the 15-digit cap is an overflow guard that only rejects typos far larger than any real step budget, so use `off`, not an extremely large integer, to disable auto-heal. `scripts/context-budget-monitor.sh` evaluates a step count against it. | `/orchestrate` |
| `max_parallel_units` | `3` | Concurrency cap on simultaneously in-flight execution units, per spec (REQ-F1.8). Resolved via `config-get` through the overlay layers; an absent key keeps the default, `0` is honored as a paused spec (no new dispatches), and the meta-tower's selector (`scripts/orchestrate-meta-select.sh`, which honors this per-spec cap alongside the fleet bound) warns and falls back to the default on a malformed value — anything other than `0` or a non-zero integer with no leading zeros (the leading-zero exclusion keeps shell arithmetic from reading the value as octal). | `/orchestrate`, `scripts/orchestrate-meta-select.sh` |
| `fleet_max_parallel_units` | `3` | Fleet-level concurrency bound for the meta-tower ("tower of towers", `/orchestrate --meta`, D-6, REQ-D1.5): the maximum number of in-flight execution units summed across **all** specs the meta-tower supervises — distinct from `max_parallel_units`, which caps a single spec. At or over the bound the meta-tower holds this step (a ready unit in some spec waits) rather than exceeding fleet-wide concurrency; the cap is enforced against the live cross-spec derivation, so it rebuilds from disk after any crash. The default is conservative — `3` holds total fleet concurrency to a single spec's worth, so enabling the opt-in meta-tower never multiplies load until an operator raises it. Resolved through `scripts/orchestrate-meta-select.sh` via `config-get`; an absent key keeps the default, a malformed value warns and falls back. | `/orchestrate` (`--meta`), `scripts/orchestrate-meta-select.sh` |
| `stale_lock_threshold` | `15m` | Age past which a per-spec advisory lock is treated as stale and may be broken by the next runner (D-10). | `/orchestrate`, `tasks-pr-sync` hook |
| `stale_marker_threshold` | `15m` | Age past which a runtime dispatch marker whose branch carries no commits is treated as stale, reverting the task to Ready so a dispatch that crashed before its first commit cannot wedge it In progress (orchestration-concurrency D-3). | `scripts/orchestrate-state.sh` (task-state derivation) |
| `review_sequence` | `[polish]` | Ordered review sequence `/execute-task`'s convergence phase runs: an inline list of nestable review-skill names (a *nestable* review skill is one invocable with `--nested`, e.g. `/polish`, `/self-review`). The default reproduces today's single `/polish --nested` convergence; an overlay can reorder or extend it. Resolved through `scripts/resolve-review-sequence.sh`; an entry naming an unknown or non-nestable skill is malformed under the REQ-E1.4 by-layer policy (degrade+warn for adopter/machine-local, hard-fail for repo-tracked) (D-6, REQ-D1.3). | `/execute-task` |
| `notification_channel` | `none` | Which channel the fleet attention/notification seam pushes through (D-13, REQ-E1.4) — the capability-vs-style split (REQ-A1.5): the notification *seam* is core, the channel *value* is overlay-owned. `none` (default) is pull-only (nothing is pushed; the operator reads the decision queue via `scripts/fleet-attention.sh queue`), dependency-free and behavior-preserving; `tmux-popup` pushes a multiplexer popup (multiplexer-user persona); `os-notify` an OS notification (non-terminal-user persona); `editor-toast` an editor toast the editor tails (editor-feedback persona). Resolved through `scripts/resolve-notification-channel.sh`; a value outside the enum is malformed under the REQ-E1.4 by-layer policy (degrade+warn for adopter/machine-local, hard-fail for repo-tracked). The channel adapters live in `scripts/fleet-attention.sh` (`notify`). | `scripts/fleet-attention.sh` (`notify`) |
| `fleet_daemon_pause` | `false` | The operator kill-switch for the autonomous daemon layer (fleet-autonomy D-15, REQ-F1.3): `true` pauses every autonomous daemon action the fleet introduces (cleanup, restart, throttle) without disabling fleet operation entirely — one lever for an operator debugging something unusual; task work (towers dispatching, workers executing) is unaffected. Every daemon mechanism the bundle introduces (Tasks 2–7) is contracted to check it via `scripts/fleet-daemon-gate.sh` **before** acting, and the gate fails closed: an unresolvable switch blocks the daemon action (degrade capability, never safety). Resolved through the shared knob resolver (`scripts/resolve-config-knob.sh` → `config-get`, fleet-autonomy D-22/REQ-G1.5); a value other than `true`/`false` is malformed under the REQ-E1.4 by-layer policy (degrade+warn for adopter/machine-local, hard-fail for repo-tracked). Write authorization: the resolving layers are human-owned surfaces (the repo-tracked overlay is committed, PR-reviewed config; the adopter and machine-local overlays live outside a dispatched worker's allowlisted write set), the same human-reviewed, human-installed posture `config/worker-settings.json` has — a worker's permission allowlist must not be widened to cover them. | `scripts/fleet-daemon-gate.sh` (called by every fleet daemon mechanism) |
| `tower_relaunch_backoff_base` | `60` | Base of the tower crash-loop backoff schedule, in seconds (fleet-autonomy D-4, REQ-A1.9): after `n` consecutive failed relaunches, the cron-scheduled tower-liveness watchdog (`scripts/fleet-tower-watchdog.sh`) waits `base * 2^(n-1)` seconds before the next attempt — structurally the same escalating shape as the worker crash-loop schedule (REQ-A1.4), but the tower's OWN knob, sharing no state or config with the worker one (many workers, one tower per spec; a shared counter would cross-contaminate — kickoff risk row 32). Resolved through the shared knob resolver (`scripts/resolve-config-knob.sh` → `config-get`, D-22/REQ-G1.5); anything other than a positive integer (no leading zero, ≤15 digits) is malformed under the REQ-E1.4 by-layer policy (degrade+warn for adopter/machine-local, hard-fail for repo-tracked). | `scripts/fleet-tower-watchdog.sh` |
| `tower_relaunch_disable_threshold` | `3` | Consecutive-failure threshold at which the tower-liveness watchdog disables itself (fleet-autonomy D-4, REQ-A1.9): once this many consecutive relaunch attempts have failed, the cron check stops relaunching, records a decision-queue entry (`scripts/fleet-attention.sh`), and audits the disable — escalating to a human instead of relaunch-looping on every cron tick. A tower later observed demonstrably alive (the human intervened) resets the counter and re-arms the watchdog. Same resolution and malformed-value policy as `tower_relaunch_backoff_base`. | `scripts/fleet-tower-watchdog.sh` |
| `fleet_model_execution` | `opus` | Which model the `execution` task-type tier of the fleet dispatch rule table runs (fleet-autonomy D-11, REQ-E1.1) — the `/execute-task` workhorse: judgment-heavy, routed to the strong tier. One column of the deterministic task-type-keyed table in `scripts/fleet-resource-select.sh` (`select execution` prints `<model>TAB<effort>TAB<command>`; effort and command are fixed table cells, code not config). Legal values are the stable Claude Code model aliases `fable`/`opus`/`sonnet`/`haiku` — aliases, not dated model ids, so the enum survives model releases. Resolved through the shared knob resolver (`scripts/resolve-config-knob.sh` → `config-get`, D-22/REQ-G1.5); a value outside the enum is malformed under the customization-overlay REQ-E1.4 by-layer policy (degrade+warn for adopter/machine-local, hard-fail for repo-tracked). No LLM call is ever in the resolution path (D-18); the selectable command set stays disjoint from `review_sequence`'s nestable-review-skill set (REQ-E1.2). | `scripts/fleet-resource-select.sh`, `/orchestrate` (dispatch) |
| `fleet_model_bookkeeping` | `sonnet` | Which model the `bookkeeping` task-type tier runs (fleet-autonomy D-11, REQ-E1.1) — the tower's reconcile/drain sweep pass (`/orchestrate --bookkeeping`): mechanical, mid tier. Same table, resolver, alias enum, and by-layer malformed policy as `fleet_model_execution`. | `scripts/fleet-resource-select.sh`, `/orchestrate` (dispatch) |
| `fleet_model_drain` | `sonnet` | Which model the `drain` task-type tier runs (fleet-autonomy D-11, REQ-E1.1) — the read-only gate-evaluation pass (`/drain`): the lightest tier (fixed `low` effort cell). Same table, resolver, alias enum, and by-layer malformed policy as `fleet_model_execution`. | `scripts/fleet-resource-select.sh`, `/orchestrate` (dispatch) |
| `fleet_throttle_default_hold` | `300` | The bounded fallback hold, in seconds, when `scripts/fleet-throttle.sh observe` detects Claude Code's native rate-limit signal but cannot parse its reset time (fleet-autonomy D-12, REQ-E1.3; kickoff risk row 24). The rate-limit prompt is version-sensitive UI text, so an unparseable rendering degrades to this hold with a warning — never an indefinite pause (an 8-day sanity ceiling bounds every engagement: the engage CLI refuses a beyond-ceiling epoch as a caller bug, and the shared path clamps), never an immediate resume, never an opaque halt. Bounded and short is the safe direction: if the limit still holds after the hold expires, the native signal re-fires and re-engages. Resolved through the shared knob resolver (`scripts/resolve-config-knob.sh` → `config-get`, D-22/REQ-G1.5); a non-positive-integer value is malformed under the customization-overlay REQ-E1.4 by-layer policy. | `scripts/fleet-throttle.sh` (`observe`) |
| `require_signed_tags` | `auto` | Signing policy for the release tag (autopilot-reflex D-4, REQ-D1.4). `auto` (default) signs when the repo has signing configured (`git user.signingkey`), otherwise creates an annotated **unsigned** tag with a warning; `require` refuses to tag unless signing is configured and succeeds; `never` always creates an unsigned annotated tag. Signing is delegated entirely to the repo's git config (`gpg.format`, `user.signingkey`, `gpg.ssh.program`) — no signer is named in the script and no signing material enters CI (a signed tag is verified with `git tag -v` before push). planwright's own repo sets `require` via its repo-tracked overlay ([`.claude/planwright.yml`](../.claude/planwright.yml)). See [`doctrine/release-tagging.md`](../doctrine/release-tagging.md) and [`docs/getting-started.md`](getting-started.md) ("Releasing planwright"). | `scripts/release-publish.sh` |
| `version_file` | `.claude-plugin/plugin.json::$.version` | Where the version of truth lives (autopilot-reflex D-13, REQ-D1.6), as `<path>::<selector>`. The selector is `$.<key>` for a top-level JSON string field (extracted with `jq`, the key passed as **data** via `--arg`, never executed) or absent for a whole-file plain `VERSION` file. The `::` separator (not `#`) avoids `config-get`'s YAML `#comment` stripping. The default reads `$.version` from the plugin manifest; override it to point at another top-level JSON string field (e.g. a `package.json`'s `$.version`) or, with no selector, a plain `VERSION` file. | `scripts/release-publish.sh`, `scripts/release-pending.sh` |
| `instruction_budget_skill_warn` | `3000` | Warn floor for a `skills/*/SKILL.md` body's word count, the secondary per-file signal (prompt-hygiene D-1, D-2). Boundary-inclusive (`>=`). A body at or over this but under the error floor is reported as a warning without failing the check. Grounded in the 500-line SKILL.md ceiling × measured `skills/` density (~8.5 words/line), recorded in prompt-hygiene D-2. | `scripts/check-instructions.sh` |
| `instruction_budget_skill_error` | `4250` | Error floor for a `skills/*/SKILL.md` body's word count (prompt-hygiene D-1, D-2). Boundary-inclusive (`>=`). A body at or over this fails the check unless a permanent exemption or a transitional `pending-diet` allowance covers it (REQ-B1.3). Derived from the 500-line ceiling × `skills/` density. | `scripts/check-instructions.sh` |
| `instruction_budget_doctrine_warn` | `2500` | Warn floor for a `doctrine/*.md` rule-doc word count (prompt-hygiene D-1, D-2). Boundary-inclusive (`>=`). `doctrine/README.md` is excluded from the per-file walk (an index, REQ-A1.1). Grounded in the 500-line ceiling × measured `doctrine/` density (~7.9 words/line). | `scripts/check-instructions.sh` |
| `instruction_budget_doctrine_error` | `4000` | Error floor for a `doctrine/*.md` rule-doc word count (prompt-hygiene D-1, D-2). Boundary-inclusive (`>=`). Over-floor fails the check unless a permanent exemption or transitional `pending-diet` allowance covers it (REQ-B1.3). | `scripts/check-instructions.sh` |
| `instruction_budget_startload_warn` | `8000` | Warn threshold for a skill's mandatory-at-start load — the SKILL.md body plus every rule doc its doctrine manifest marks `run-start` — the **primary** budget (prompt-hygiene D-1). Boundary-inclusive (`>=`). What degrades instruction-following is total instruction context per invocation, so a fat run-start doc charges every skill that front-loads it. | `scripts/check-instructions.sh` |
| `instruction_budget_startload_error` | `10000` | Error threshold for a skill's mandatory-at-start load (prompt-hygiene D-1). Boundary-inclusive (`>=`). A start-load offender can be carried only by a transitional `pending-diet` allowance, never a permanent exemption (REQ-B1.3); its fix is a diet or point-of-use reclassification (REQ-D1.1). | `scripts/check-instructions.sh` |
| `instruction_budget_closure_warn` | `15000` | Warn threshold for a skill's reachable closure — the start-load plus every rule doc its manifest defers to `point-of-use` (prompt-hygiene D-1). Boundary-inclusive (`>=`). The loose budget: point-of-use bulk is read only when its path is taken. | `scripts/check-instructions.sh` |
| `instruction_budget_closure_error` | `20000` | Error threshold for a skill's reachable closure (prompt-hygiene D-1). Boundary-inclusive (`>=`). A closure offender can be carried only by a transitional `pending-diet` allowance, never a permanent exemption (REQ-B1.3b); its fix is a content diet. | `scripts/check-instructions.sh` |
| `instruction_budget_injected_warn` | `200` | Warn floor for the **static** injected prose a `hooks.json`-registered hook contributes via `additionalContext` / `hookSpecificOutput` (prompt-hygiene REQ-A1.4, REQ-B1.7, D-13). Boundary-inclusive (`>=`). Warn-only: this partly-dynamic per-session surface is always reported as an audit row but **never fails** the check (a hook whose static prose cannot be extracted is a parse-failure warning, not an error). Grounded in the ~40-word present surface plus headroom. | `scripts/check-instructions.sh` |
| `instruction_manifest_completeness_required` | `true` | When `true`, every `skills/*/SKILL.md` must declare a doctrine manifest; a manifest-less skill is an error (prompt-hygiene REQ-A1.2, D-3). Wired at Task 3, when the manifests land — it stops a future manifest-less skill from silently under-reporting its start-load. Distinct from the scoring rule (a manifest-less skill is still scored body-only) and the malformed-manifest error (a present-but-non-boolean value here is itself fail-loud, REQ-B1.8). Absent in every layer it defaults off, so an adopter not yet on the manifest convention is not forced into it. | `scripts/check-instructions.sh` |
